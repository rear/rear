
# Using ReaR with iSCSI Devices

This section describes how to use Relax-and-Recover on systems where the
operating system resides on iSCSI storage, including systems that have no
local hard drives at all.

## Overview

iSCSI (Internet Small Computer Systems Interface) allows block-level storage
to be accessed over a standard TCP/IP network. A system that boots and runs
entirely from iSCSI storage has no local disks; all persistent storage is
provided by a remote iSCSI target.

Because the ReaR rescue/recovery system must be able to reach and log in to
the iSCSI target before it can recreate or restore the system, several extra
steps are required compared to a system with local storage:

1. The iSCSI initiator programs and configuration must be included in the
   rescue system.
2. The rescue system must set up the network interface used for iSCSI and log
   in to the iSCSI target before `rear recover` can run.
3. ReaR must be told that the system boots over SAN/iSCSI so that the
   relevant multipath and SAN executables are included.

## Prerequisites

- The `open-iscsi` package (providing `iscsid` and `iscsiadm`) must be
  installed and configured on the source system.
- The iSCSI target must be reachable from the recovery environment using the
  same network interface and IP address as on the running system.
- The iSCSI initiator must already be logged in and the sessions recorded in
  `/var/lib/iscsi` on the source system (i.e. `iscsiadm -m node` should list
  the target before you run `rear mkrescue`).

## Configuration

Add the following settings to `/etc/rear/local.conf`.

### Include iSCSI programs and configuration

```bash
# Include the iSCSI initiator daemon and administration utility
REQUIRED_PROGS+=( iscsid iscsiadm )

# Copy the iSCSI configuration and persistent session data into the rescue system
COPY_AS_IS+=( /etc/iscsi /var/lib/iscsi )
```

`/etc/iscsi` holds the initiator configuration (including `initiatorname.iscsi`
and `iscsid.conf`) while `/var/lib/iscsi` holds the discovered target records
that `iscsiadm` needs to log in automatically.

### Enable SAN boot support

```bash
# Include multipath and SAN-related executables needed for SAN-only systems
BOOT_OVER_SAN=y
```

If the system uses multipath on top of iSCSI, also set:

```bash
AUTOEXCLUDE_MULTIPATH=n
```

### Set up networking and log in to the iSCSI target automatically

The rescue system must bring up the iSCSI network interface and log in to the
target before `rear recover` attempts to access the disk. Use
`NETWORKING_PREPARATION_COMMANDS` to run these steps automatically at boot
time.

Replace the IP address, prefix length, and interface name with values that
match your environment:

```bash
NETWORKING_PREPARATION_COMMANDS=(
    'ip addr add 192.168.111.222/24 dev enp6s0'
    'ip link set dev enp6s0 up'
    'sleep 1'
    'mkdir -p /run/lock/iscsi'
    'iscsid'
    'sleep 1'
    'iscsiadm -m node --loginall=automatic'
    'return'
)
```

The `return` statement at the end causes the network setup script to exit
immediately after these commands, preventing the autogenerated network setup
code from running and potentially overwriting the configuration you set
manually.

**Note about `mkdir -p /run/lock/iscsi`:** Some distributions (notably
RHEL 9 and derivatives) require the directory `/run/lock/iscsi` to exist
before `iscsid` can start. Because this directory lives under `/run` (a
tmpfs), it is not preserved across reboots and must be created explicitly
inside the rescue system.

### Complete `local.conf` example

```bash
OUTPUT=ISO
BACKUP=NETFS
BACKUP_URL=nfs://192.168.111.1/path/to/rear/backup

REQUIRED_PROGS+=( iscsid iscsiadm )
COPY_AS_IS+=( /etc/iscsi /var/lib/iscsi )

BOOT_OVER_SAN=y
# Uncomment if using multipath over iSCSI:
# AUTOEXCLUDE_MULTIPATH=n

NETWORKING_PREPARATION_COMMANDS=(
    'ip addr add 192.168.111.222/24 dev enp6s0'
    'ip link set dev enp6s0 up'
    'sleep 1'
    'mkdir -p /run/lock/iscsi'
    'iscsid'
    'sleep 1'
    'iscsiadm -m node --loginall=automatic'
    'return'
)
```

## Step-by-step procedure

### 1. Prepare the source system

Ensure that the iSCSI session is active and that target records exist in
`/var/lib/iscsi`:

```bash
iscsiadm -m session
iscsiadm -m node
```

### 2. Create the rescue image

```bash
rear -v mkrescue
```

or, if you also want ReaR to create the backup archive:

```bash
rear -v mkbackup
```

### 3. Verify the rescue system (optional but recommended)

Before attempting a full recovery on production hardware, boot the rescue
image on a test system and verify that the iSCSI disks become available:

```bash
iscsiadm -m session          # should list an active session
lsblk                        # should show the iSCSI block device(s)
```

Do **not** run `rear recover` during this verification step unless you
intend to overwrite the data on the iSCSI disks.

### 4. Recover the system

Boot the rescue image on the target hardware. After the rescue system starts
and `NETWORKING_PREPARATION_COMMANDS` has run, the iSCSI disks should be
visible. Then start the recovery:

```bash
rear recover
```

## Manual iSCSI login (without NETWORKING_PREPARATION_COMMANDS)

If you prefer to log in to the iSCSI target manually after booting the rescue
system, run the following commands at the rescue shell prompt:

```bash
mkdir -p /run/lock/iscsi
iscsid
sleep 1
iscsiadm -m node --loginall=automatic
```

Then verify that the disk is visible (`lsblk`) before running `rear recover`.

## Advanced: using a pre-recovery script

For more complex environments, such as those with multiple iSCSI targets or
where the IP address must be chosen at recovery time, you can use a custom
pre-recovery script instead of `NETWORKING_PREPARATION_COMMANDS`.

Create the script at `/etc/rear/pre-recovery-script.sh` on the source system
and include it in the rescue image:

```bash
COPY_AS_IS+=( /etc/rear/pre-recovery-script.sh )
PRE_RECOVERY_SCRIPT=/etc/rear/pre-recovery-script.sh
```

The script can prompt the operator for the network address and iSCSI target
IQN and then perform the login steps above.

## Troubleshooting

**`iscsid` fails to start — "cannot open /run/lock/iscsi"**

Create the directory first:

```bash
mkdir -p /run/lock/iscsi
iscsid
```

**No iSCSI block devices visible after login**

Check that the session is established:

```bash
iscsiadm -m session
```

If no sessions are listed, try a manual discovery and login.  If the
target nodes were previously configured for automatic startup
(`node.startup = automatic` in `iscsid.conf` or set via `iscsiadm`),
use:

```bash
iscsiadm -m discovery -t sendtargets -p <target-IP>
iscsiadm -m node --loginall=automatic
```

If the nodes are not configured for automatic startup, log in to a
specific target by IQN and portal instead:

```bash
iscsiadm -m discovery -t sendtargets -p <target-IP>
iscsiadm -m node --targetname <iqn.xxxx-xx.com.example:target> \
         --portal <target-IP>:<port> --login
```

**ReaR rescue image is too large**

If the rescue image size exceeds the capacity of your boot medium after adding
the iSCSI files, review the contents of `/var/lib/iscsi` and
`/etc/iscsi` and exclude unnecessary files via `COPY_AS_IS_EXCLUDE`.

**Kernel does not load automatically**

Ensure that `OUTPUT=ISO` is set (or another output method appropriate for
your boot medium) and that the kernel and initrd are being generated without
errors. Run `rear -v mkrescue` and check the log at
`/var/log/rear/rear-<hostname>.log`.
